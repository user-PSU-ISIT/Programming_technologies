# Лабораторная работа №1 - Первичная настройка проекта

В рамках первой лабораторной работы необходимо выполнить базовую настройку проекта (**frontend + backend**) с соблюдением современных стандартов разработки.

---

## Критерии выполнения

Лабораторная работа считается выполненной при соблюдении следующих условий:

### 1. Инструменты качества кода

- Настроен и подключён линтер
- Настроен и подключён форматтер кода
- Правила линтинга и форматирования применяются ко всему проекту
- Добавлена конфигурация игнорируемых файлов (`.eslintignore`, `.prettierignore` или `.gitignore`)

---

### 2. Реализация приложения

- Реализован REST API с соблюдением принципов REST:
  - корректное использование HTTP-методов (`GET`, `POST`, `PUT`/`PATCH`, `DELETE`)
  - корректные HTTP-статусы
  - единообразная структура ответов
  - корректная обработка ошибок

- Минимальный стек frontend: **React + TypeScript + Vite**.
- Стек backend выбирается студентом самостоятельно.\
<!-- - Реализовано несколько CRUD-операций. -->

---

### 3. Документация

В проекте должен присутствовать `README.md` (отдельно для frontend и backend), содержащий:

- Краткое описание проекта
- Описание файловой структуры проекта
- Инструкцию по установке и запуску
- Список используемых технологий с ссылками на официальную документацию
- Описание переменных окружения (если используются)
- Описание неочевидных технических решений или особенностей реализации

---

### 4. Настроенный pre-commit

- В проекте должен быть настроен `pre-commit`
- Коммит **не должен выполняться**, если:
  - В репозитории присутствуют файлы, превышающие допустимый размер
  - В коде обнаружены маркеры неразрешённых конфликтов
  - Код не соответствует правилам линтера или форматтера
  - Присутствуют неиспользуемые импорты, переменные или «мёртвый» код
  - Названия файлов не соответствуют принятому в проекте стандарту (например: `kebab-case`, `camelCase`, `PascalCase` — стандарт должен быть явно определён)

> Допускается автоматическая очистка кода, однако коммит не должен выполняться без повторной проверки разработчиком.

---

### 5. Переменные окружения (при наличии)

- Использование `.env`
- Разделение конфигураций для `development` и `production`

---

### 6. Docker

- Наличие `Dockerfile` для frontend и backend.
- Наличие `docker-compose` для локального запуска проекта.

---

### 7. Код-стайл

- Единый стиль именования:
  - переменных,
  - функций,
  - компонентов,
  - файлов

- Описание принятых соглашений в `README.md`
