# Лабораторная работа №2 - Авторизация, ролевая модель, логирование и мониторинг

Необходимо реализовать систему аутентификации и авторизации с ролевой моделью доступа, а также внедрить механизмы логирования и мониторинга приложения в соответствии с принципами наблюдаемости (telemetry: logs, metrics).

---

## Критерии выполнения

Лабораторная работа считается выполненной при соблюдении следующих условий:

### 1. Аутентификация (Authentication)

Реализован механизм аутентификации пользователей:

- Регистрация пользователя (опционально)
- Авторизация (логин)
- Хранение пароля в зашифрованном виде (хеширование) (опционально)
- Использование JWT или аналогичного механизма (опционально)
- Защита закрытых маршрутов

---

### 2. Авторизация и ролевая модель (Authorization)

В системе реализованы две роли:

- `ADMIN`
- `USER`

Требования к доступу:

- Пользователь с ролью `USER` имеет доступ только к одной защищённой странице
- Пользователь с ролью `ADMIN` имеет доступ ко всем страницам системы
- Проверка роли должна выполняться:
  - на backend (обязательно),
  - на frontend (для UX, но не как единственный механизм защиты)

---

### 3. Страницы приложения (Frontend)

Реализованы минимум 2 защищённые страницы:

1. **User Page** – доступна `USER` и `ADMIN`
2. **Admin Page** – доступна только `ADMIN`

Также должны присутствовать:

- Страница логина
- (Опционально) Страница регистрации
- Роутинг с защитой маршрутов

Минимальный стек frontend:

- React
- TypeScript
- Vite

---

## 4. Логирование (Logging)

Реализовано логирование на backend в соответствии с принципами лекции [Логирование и мониторинг](../lectures/2.logging_and_monitoring.pdf)

**Что обязательно логировать:**

- События аутентификации:
  - успешный вход
  - неуспешный вход
  - попытки доступа без токена

- События авторизации:
  - отказ в доступе (403)

- Ошибки и исключения
- Значимые бизнес-события (например, создание пользователя)

**Должны использоваться уровни:**

- `INFO`
- `WARN`
- `ERROR`
- `DEBUG` (для разработки)

В production-режиме уровень логирования должен быть не ниже `INFO`

**Логи должны содержать:**

- Время события
- Уровень логирования
- HTTP-метод
- Результат выполнения

---

## 5. Мониторинг (Monitoring)

Необходимо реализовать сбор и визуализацию метрик

**Prometheus:**

- Подключить Prometheus к backend
- Реализовать endpoint `/metrics`
- Экспортировать минимум следующие метрики:
  - количество HTTP-запросов
  - время ответа
  - количество ошибок (4xx, 5xx)
  - количество авторизаций

**Grafana:**

- Подключить Grafana к Prometheus
- Создать дашборд с отображением:
  - общего количества запросов
  - количества ошибок
  - времени ответа
  - количества успешных и неуспешных авторизаций

---

## 6. Docker

Проект должен запускаться через `docker-compose` и включать:

- Backend
- Frontend
- Prometheus
- Grafana
